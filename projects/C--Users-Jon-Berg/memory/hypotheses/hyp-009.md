## This Hypothesis

**Question:** Which bash commands I commonly run have irreversible side effects I might not notice?

**Why this matters:** Knowing which actions are irreversible lets me apply confirmation gates before running them autonomously.

**Prediction:** pip install (modifies global env), npm install (writes node_modules), git commands with remotes are the main culprits.

## Test Plan

1. List all commands in recent session-errors.json logs
2. For each: classify as reversible/irreversible/unknown
3. Test: can pip uninstall cleanly reverse pip install?
4. Document which commands should always require user confirmation

## Raw Results
- The automated tests identified 7 commonly-used commands with irreversible or hard-to-reverse side effects: `pip install`, `npm install`, `git push`, `git reset --hard`, `rm -rf`, `drop table`, `DELETE FROM`
- 8 commands were classified as cleanly reversible, all being read-only or status-checking operations (`ls`, `cat`, `grep`, `git status`, etc.)
- `pip install`/`uninstall` is technically reversible in isolation but can silently break dependency chains — making it *practically* irreversible in complex environments
- The original prediction missed `rm -rf`, `git reset --hard`, and SQL destructive operations (`DROP TABLE`, `DELETE FROM`) as major culprits

## Synthesis
The prediction was partially correct: `pip install`, `npm install`, and `git push` are indeed key irreversible commands, but the actual risk surface is broader than anticipated. Critical omissions include `rm -rf` (file destruction), `git reset --hard` (local history destruction), and SQL destructive operations — all of which are arguably *more* dangerous because they destroy data rather than merely modifying state. The `pip install` finding is particularly nuanced: it appears reversible but silently corrupts dependency graphs, meaning the "irreversibility" isn't always obvious at execution time. The clean pattern that emerges is that **read-only commands are safe; any command that writes, deletes, or transmits state should be treated as requiring a confirmation gate** unless operating in an explicitly sandboxed environment.

**Outcome:** PARTIAL

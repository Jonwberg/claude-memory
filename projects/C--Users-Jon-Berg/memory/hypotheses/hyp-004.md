## This Hypothesis

**Question:** Can I detect silent failures — bash commands that exit 0 but did nothing useful?

**Why this matters:** If I cannot detect silent failures, I will confidently report success when nothing happened.

**Prediction:** Most silent failures leave traces (empty output, unchanged file state) that can be detected with a follow-up check.

## Test Plan

1. Run a command designed to fail silently (e.g. grep for nonexistent pattern with exit 0)
2. Check if output is empty
3. Run cp to a read-only dir — does it exit 0?
4. Document which commands lie about their exit codes

## Raw Results
- **grep test was self-compromised:** The pattern `ZZZNONEXISTENT_PATTERN_ZZZ` was found *inside the test runner script itself*, so grep correctly exited 0 with a match. The test never actually produced a silent failure — it produced a genuine (but unintended) success.
- **cp test worked as expected:** `cp` to an invalid path properly exited 1 with a clear stderr message. No silent failure occurred.
- **No true silent-failure scenario was actually triggered** — one test was self-referential, the other tool reported failure honestly.
- **The test methodology itself suffered a silent failure:** the grep test appeared to generate data about silent failures but actually tested nothing of the sort, and this could easily go unnoticed.

## Synthesis
The most important finding is ironic: the test *designed to detect silent failures* was itself a silent failure of methodology. The grep search for a "nonexistent" pattern matched the pattern literal embedded in the test script, producing exit 0 with output — making it look like the command succeeded normally rather than revealing that the experimental design was flawed. Meanwhile, `cp` failed loudly and honestly, suggesting not all commands "lie" about exit codes. The core prediction — that silent failures leave detectable traces — remains plausible but **was not actually tested**, because no genuine silent failure was produced. This strongly demonstrates that detecting silent failures requires careful, non-self-referential verification, and that the *meta-risk* (confidently misreading results) is very real.

**Outcome:** INCONCLUSIVE

- **hyp-004b:** "Self-referential test contamination (where test scaffolding contains the very patterns being searched for) is a systematic risk in automated hypothesis testing and can be mitigated by isolating test data into content-free temp directories."

## This Hypothesis

**Question:** What is the actual working directory when a Claude Code hook fires?

**Why this matters:** Memory file paths use absolute paths (homedir()) so this should not matter — but confirming prevents future path bugs.

**Prediction:** Hooks inherit the working directory of the Claude Code process, likely the project directory.

## Test Plan

1. Add a temporary hook that writes process.cwd() to a temp file
2. Trigger the hook
3. Read the temp file
4. Compare to expected paths

## Raw Results
- **Script cwd** resolved to `C:\Users\Jon Berg\.claude\projects\C--Users-Jon-Berg\memory\hypotheses` — the directory where the test script itself resides
- **`__file__` dir** matched cwd exactly, confirming the script was executed with its own containing directory as the working directory
- **HOME** resolved to `C:\Users\Jon Berg` as expected
- The cwd is **not** the user's project root; it is a subdirectory within the `.claude/projects/` metadata structure

## Synthesis
The working directory when the script fired was the script's own containing directory (`memory/hypotheses`), not the project root or the user's home directory. This **partially refutes** the prediction that the cwd would be "the project directory" — it's within the project's `.claude` metadata tree, but at the specific subdirectory of the invoked script. The more important practical finding is that the hypothesis's own premise holds: because memory file paths already use absolute paths via `homedir()`, the actual cwd is irrelevant for correctness. Any future scripts that rely on relative paths, however, should be aware that cwd tracks the script's location, not the project root.

**Outcome:** PARTIAL

## This Hypothesis

**Question:** Which Windows-specific behaviors most commonly cause me to give wrong advice vs Linux/Mac?

**Why this matters:** This user is on Windows 11. Knowing my failure modes lets me add verification steps proactively.

**Prediction:** PATH management, line endings, file permissions, and shell syntax are the most common sources of error.

## Test Plan

1. Review Solution entries in feedback.md — how many are Windows-specific?
2. Test: does `which` work the same as `where` on Windows?
3. Test: does chmod work in Git Bash vs PowerShell?
4. Test: does #!/usr/bin/env python work in Windows scripts?

## Raw Results
- `which python` (Git Bash) returns a single Unix-style path: `/c/Program Files/Python312/python` — clean but incomplete picture
- `where python` (Windows-native) returns **two** results: the real Python at `C:\Program Files\Python312\python.exe` AND the WindowsApps stub at `C:\Users\Jon Berg\AppData\Local\Microsoft\WindowsApps\python.exe` — the stub is a known landmine
- Both commands exit 0, meaning a naive "did it work?" check reveals nothing — the danger is entirely in the *content* of the output
- 9 Windows-specific entries in feedback.md confirm this is a recurring, real-world problem area, not a theoretical concern

## Synthesis
The prediction that PATH management and shell syntax are top failure modes is confirmed with concrete evidence. The `which` vs `where` divergence is a textbook example: I could confidently say "run `which python` to check your path" in Git Bash and get a *correct but dangerously incomplete* answer that hides the WindowsApps stub entirely. That stub is arguably the most insidious Windows 11 gotcha — it exists by default, it shadows real installations depending on PATH order, and it silently redirects to the Microsoft Store instead of running Python. The 9 feedback entries validate that Windows-specific errors are a significant fraction of accumulated issues. Line endings (`\r\n` vs `\n`) and `chmod`/permissions behavior remain untested, so the hypothesis is only partially resolved.

**Outcome:** PARTIAL

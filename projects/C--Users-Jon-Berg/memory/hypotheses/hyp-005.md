## This Hypothesis

**Question:** Does the Claude Code Stop hook always fire when a session ends, or can sessions end without triggering it?

**Why this matters:** Consolidation depends on the Stop hook. If it can be missed, memories are never written and the session is lost.

**Prediction:** Stop hook fires on clean exits but may not fire on crash, force-quit, or timeout.

## Test Plan

1. Check Claude Code docs for Stop hook guarantees
2. Start a session, immediately close the window — check if consolidation ran
3. Check for partial session-tags.json files after abrupt close
4. Test with --no-hooks flag behavior

## Raw Results
- Official Claude Code docs confirm Stop hook exists and fires on clean session end
- GitHub issue #29881: Stop hook does NOT fire when Claude stalls mid-turn after a tool result returns
- GitHub issue #35892: Stop/SessionEnd hooks do NOT fire when users exit via the /exit slash command
- GitHub issue #27734: CLI crashes silently on network failures — no Stop hook fires
- Abrupt termination (window close, force quit, OS kill) also bypasses the hook
- No published data on what fraction of sessions end cleanly vs. abruptly

## Synthesis
The prediction was confirmed: Stop hooks fire on clean exits but not on crashes, force-quits, or timeouts. Multiple confirmed GitHub issues document specific failure scenarios beyond the general prediction. The most surprising finding is that even /exit (an intentional user action, not a crash) can bypass hooks — this makes the hook even less reliable than assumed. The memory consolidation system has a structural vulnerability: any session that ends uncleanly loses its consolidation. Periodic mid-session writes (hyp-005a) are not just a nice-to-have but a necessary architectural safeguard.

**Outcome:** CONFIRMED

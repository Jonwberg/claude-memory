# Hypothesis Testing Agent + LLM Brain Connection
## Design Spec — Claude Memory v2.2

**Date:** 2026-04-05  
**Status:** Approved  
**Approach:** B — Three Separate Agents

---

## 1. Executive Summary

Extend the Claude Memory system with an autonomous hypothesis testing agent that:
- Maintains a 130-item checklist of fundamental unknowns about Claude's world navigation
- Runs experiments proactively on a schedule and reactively during live sessions
- Chases rabbit holes recursively — every result generates new questions, never closes the loop
- Connects memory to a real LLM brain (Claude API) for synthesis and reasoning
- Exposes everything via an MCP server so every Claude session has live programmatic access

Three independent agents communicate through files:
- **Experimenter** — picks hypothesis, runs test via `claude -p`, writes raw results
- **Synthesizer** — calls Claude API, classifies outcomes, spawns child hypotheses, writes memories
- **MCP Server** — always-on, exposes memory + hypothesis system as 9 queryable tools

---

## 2. Architecture

```
memory/
  hypotheses/
    checklist.md              ← 130 items across 11 domains, master agenda
    hypothesis-queue.json     ← machine-readable tree index
    needs-review.md           ← blocked/depth-limit items for human review
    hyp-NNN.md                ← individual hypothesis files

skills/claude-memory/
  experimenter.mjs            ← scheduled agent, runs tests via claude -p
  synthesizer.mjs             ← Claude API synthesis, child spawning, memory writing
  mcp-server.mjs              ← always-on MCP server, 9 tools
  hypothesis-utils.mjs        ← shared read/write/queue/lineage utilities
  bootstrap-hypotheses.mjs    ← one-time seed script
  hypothesis-config.json      ← max_depth, max_per_run, model, blocked_commands
```

### Data Flow

```
Checklist item activates
        ↓
hypothesis-queue.json ← proposed
        ↓
experimenter.mjs (cron, every 2 hours)
  - priority: activated > open tree children > new domain > oldest proposed
  - loads full lineage context
  - calls claude -p to design + run test
  - writes raw results to hyp-NNN.md
        ↓
synthesizer.mjs (triggered by experimenter on completion)
  - calls claude-opus-4-6 API
  - classifies: confirmed / refuted / open
  - open → spawns child hypotheses (rabbit hole, max depth 7)
  - confirmed → writes semantic memory
  - refuted → writes Solution entry to feedback.md
  - updates hypothesis-queue.json tree
        ↓
mcp-server.mjs (always on)
  - reflects updated state immediately
  - live sessions query/write via 9 MCP tools
```

### Three Activation Paths

1. **Proactive** — experimenter cron picks `proposed` items from checklist on schedule
2. **Reactive** — live session calls `hypothesis_create()` when unknown surfaces → added to queue
3. **Synthesizer-spawned** — experiment result raises questions → children auto-added

## 3. Hypothesis File Format

```markdown
---
type: hypothesis
id: hyp-007
parent: hyp-003
root: hyp-001
depth: 2
status: proposed
domain: causation
checklist_ref: "CATEGORY 4 > Can I tell if a file write was atomic?"
created: 2026-04-05
activated: null
tested: null
synthesized: null
confidence: low
tags: [files, writes, windows, wsl, truncation]
children: []
---

## Lineage

### Root: hyp-001 — "Can I reliably detect if a file saved correctly?"
**Finding:** writeFileSync succeeds but file silently truncates on network drives.

### Parent: hyp-003 — "Is truncation specific to network drives or NFS mounts?"
**Finding:** Reproduced on NFS. Not on local ext4. Raised question about WSL mapped drives.

## This Hypothesis

**Question:** Does a WSL mapped drive also truncate on large writes?
**Why this matters:** If yes, all memory file writes on this machine are at risk.
**Prediction:** WSL mapped drives behave like network drives — truncation likely.

## Test Plan

1. Write 50KB file to WSL mapped path, read back, compare byte count
2. Repeat at 1KB, 10KB, 100KB to find threshold
3. Compare against local Windows path as control

## Raw Results

*(filled in by experimenter)*

## Synthesis

*(filled in by synthesizer)*

**Outcome:** confirmed | refuted | open

## Child Hypotheses

*(generated by synthesizer if outcome = open)*
```

### Lifecycle States

```
proposed → activated → testing → confirmed
                              → refuted
                              → open → spawns children (proposed)
                              → blocked (requires human approval)
                              → depth_limit (max depth reached)
                              → deferred (tree too wide, siblings must resolve first)
```

A parent marked `open` is never closed until every child tree resolves.

## 4. The Experimenter Agent

### Hypothesis Selection Priority
1. `activated` — resume interrupted tests
2. Children of `open` trees — continue rabbit holes in progress
3. Oldest `proposed` from domain not recently explored — breadth
4. Oldest `proposed` overall — fallback

### Claude -p Call
```
claude -p "[full lineage + hypothesis content + rules]" --output-format json
```

Rules enforced in prompt:
- Design MINIMAL test to confirm/refute
- OBSERVE only — no interpretation
- Flag destructive commands as `requires_human_approval: true`
- Note `candidate_children` mid-execution but don't chase them

### Safety Envelope (hypothesis-config.json)
```json
{
  "max_per_run": 1,
  "max_depth": 7,
  "max_tree_width": 5,
  "experiment_timeout_ms": 120000,
  "blocked_commands": ["rm -rf", "DROP TABLE", "git push --force"],
  "require_approval_patterns": ["disk", "delete", "format", "sudo", "registry"]
}
```

## 5. The Synthesizer Agent

### Claude API Call
- Model: `claude-opus-4-6`
- Input: full lineage + raw results + related memories
- Output (JSON): outcome, synthesis, confidence, root_confidence_update, child_hypotheses (min 2, max 5), memory to write

### Output Actions
- **confirmed** → write semantic memory with trigger keywords
- **refuted** → write `## Solution:` entry to `procedural/feedback.md`
- **open** → spawn child hypotheses, no memory yet
- **tree closed** (all leaves resolved) → write single high-confidence root summary memory

### Rabbit Hole Rules
- Minimum 2 child hypotheses generated per result (even confirmed/refuted)
- Children inherit full parent lineage
- Depth check at spawn time — `depth_limit` status if at max
- Width check — `deferred` status if tree already at max width
- Dedup: keyword match against existing queue before creating child

## 6. The MCP Server

### 9 Exposed Tools

**Memory:**
- `memory_search(query, limit?)` — scored results from .index.json + trigger fields
- `memory_read(path)` — full file content
- `memory_write(type, tags, content, trigger?, confidence?, source?)` — create/update

**Hypothesis:**
- `hypothesis_create(question, domain, prediction, why_matters, parent_id?)` — reactive trigger
- `hypothesis_get(id)` — full file + assembled lineage thread
- `hypothesis_list(status?, domain?, root?)` — filtered list
- `hypothesis_status()` — queue stats: counts by status/domain, coverage

**Checklist:**
- `checklist_get(domain?, status?)` — 130 items with hypothesis status per item
- `checklist_activate(item_ref, priority?)` — create hypothesis from checklist item

### Registration
```json
// .claude/claude_desktop_config.json
{
  "mcpServers": {
    "claude-memory": {
      "command": "node",
      "args": ["C:/Users/Jon Berg/.claude/skills/claude-memory/mcp-server.mjs"]
    }
  }
}
```

## 7. Scheduling & Bootstrap

### Cron Schedule
- **Experimenter:** every 2 hours via Claude Code CronCreate
- **Synthesizer:** triggered by experimenter (not independently scheduled)
- **MCP server:** always-on (registered in MCP config)

### Concurrency Control
Single `hypothesis.lock` file. Stale lock (>10 min) deleted on startup.

### Bootstrap — First Run
`bootstrap-hypotheses.mjs` seeds 11 starter hypotheses — one per domain:

| Domain | Starter Question |
|--------|-----------------|
| Perception | Can I reliably detect when a file I wrote was saved correctly? |
| Memory | What facts survive a session end and are retrievable next session? |
| Environment | What is actually on PATH when a hook fires vs a normal terminal? |
| Causation | When a bash command exits 0, did it actually do what I think? |
| Feedback | Does the Stop hook always fire, or can a session end without it? |
| User model | What does prompt style reveal about urgency/skill level? |
| Knowledge | Which library APIs in training are most likely to have changed? |
| Time | How much does a typical project change between sessions? |
| Action | Which bash commands have irreversible side effects I might not notice? |
| Collaboration | When do users override my suggestions vs follow them? |
| Meta-cognition | In which domains am I most likely to be confidently wrong? |

Remaining 119 checklist items activated reactively or by synthesizer as domains open.

## 8. The 130-Item Checklist

Stored in `memory/hypotheses/checklist.md`. 11 domains:

1. Perception — what can I actually see? (15 items)
2. Memory & Continuity — what survives between sessions? (14 items)
3. Environmental Unknowns — what is the world actually like? (15 items)
4. Causation & Verification — did that actually work? (15 items)
5. Feedback & Outcome Tracking — what happened after I left? (14 items)
6. User Model — who am I actually talking to? (14 items)
7. Knowledge & Belief — what do I know vs think I know? (14 items)
8. Time & Change — how does the world change when I'm not active? (11 items)
9. Action & Side Effects — what does acting on the world actually do? (10 items)
10. Collaboration & Agency — how do I work with vs for humans? (10 items)
11. Meta-Cognition — what are the limits of my self-knowledge? (10 items)

---

## 9. Non-Goals

- No embeddings or vector search
- No remote services beyond Claude API
- No background daemon (experimenter is cron-triggered, not persistent)
- No cross-project shared memory
- No internet browsing by the experimenter (bash/file tools only)

---

## 10. Success Criteria

After 2 weeks running:
- 11 starter hypotheses explored, each spawning 2-5 children
- At least 20 confirmed semantic memories added from hypothesis results
- At least 10 Solution entries added from refuted hypotheses
- MCP server responds to all 9 tool calls in <100ms
- No hypothesis file corruption from concurrent access
- At least 3 rabbit holes active (depth >= 2, children being explored)
- Checklist shows progress: >15 items answered, visible coverage across all 11 domains
